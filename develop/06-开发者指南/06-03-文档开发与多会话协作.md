# 文档开发与多会话协作

## 目的与边界

本文档说明如何在编写与维护本仓库文档时，使用**多会话协作**（多个 AI 会话/助手并行分工）来提升产出效率。

重要说明：这里的“多会话协作”仅用于**文档开发流程**；与量化平台的系统架构无关，不代表系统引入新的服务拆分或通信机制。

## 推荐工作流

### 1) 拆分任务

按文档或章节拆分为多个独立任务，并行推进。例如：
- API 文档修订
- 开发者指南修订
- 部署与运维手册修订
- 测试与验证文档修订

### 2) 事实校验（必须）

每次写到以下内容时，都要用命令确认其在仓库中真实存在且可运行：
- 文件路径与目录结构（用 `ls` / `find` / `rg --files`）
- 命令行示例（至少本地跑一遍）
- 模块/类/函数名（用 `rg` 定位到源码）

原则：不写“猜出来”的路径和命令。

### 3) 统一对齐口径

所有修改过的文档应统一记录：
- 对齐基线（当前 commit）
- 最后更新日期
- 本次修改影响范围（只改文档/是否涉及代码）

## 收口检查清单（文档自检）

- 文档中的路径都存在
- 文档中的命令可以执行（至少在开发环境跑通）
- 文档不包含“平台架构拆分为多个独立代理组件/消息总线通信”等表述

## 文档顶部模板（可复制）

```markdown
## 对齐信息

- 对齐基线：<填写 git commit>
- 最后更新：YYYY-MM-DD
- 变更范围：仅文档
```
